# Codex Daily Report Skill

这个 skill 用来把本机 Codex 会话记录整理成“当天工作日报”，或把一周内已经生成的日报合并成“工作周报”。

核心目标只有一个：根据某一天、某个工作区里的真实对话，还原当天做了什么，再站在团队和管理者视角写成非技术同事也能看懂的日报。

默认数据源：

- `~/.codex/state_5.sqlite`
- `~/.codex/sessions/.../*.jsonl`
- 目标工作区及子仓库的 Git 记录、未提交改动和当天新增/修改文件
- 本机 token 统计脚本或命令（如果用户提供或能在工作区找到）
- 目标工作区里已经生成的日报 Markdown 文件（生成周报时优先使用）
- 可选 Issue/Bug tracker 数据（如果用户的本地环境已经有只读查询方式）

## 输入默认值

按下面规则确定四件事；用户未指定日期时，直接使用当前本地日期，不要追问是哪一天：

- 目标日期：用户没指定时，默认使用当前本地日期
- 目标工作区
- 总工时
- 输出位置

默认规则：

- 用户没指定工作区时，优先使用当前工作区
- 用户没指定总工时时，默认按 `8h` 生成日报
- 用户明确给了 `8h`、`10h` 之类总工时，就按这个总数拆分
- 如果用户在要求重新生成时说“都改为 Xh / 按 Xh / 总工时 Xh”，必须把 Xh 作为本次日报的唯一总工时，不能沿用旧日报里的工时
- 默认只生成“工作内容”，不要写“明日计划”；只有用户明确要求明日计划时才补

## 输出路径优先级

不要把输出目录写死到某个项目。

按下面顺序处理：

1. 用户明确给了文件路径或目录，就用用户指定的位置。
2. 用户没给时，先在当前工作区查是否已有稳定的日报目录约定，再沿用原有目录。
3. 如果当前工作区没有明显约定，默认写到当前工作区下的 `./daily/{date}.md`。

如果目标文件已经存在：

- 用户要求“删了重新生成 / 重新生成 / 生成今天日报”时，先删除目标日期已有日报文件，再从会话和 Git 重新生成，避免旧日报影响判断
- 只有用户明确要求“在现有日报基础上修改 / 调整这份日报”时，才先读取已有内容再做补充或调整
- 删除旧日报只针对目标日期的日报文件，不要清理需求文档、测试记录或其他工作产物

## 快速入口

- 线程查询、`rollout_path` 查询、`jq` 抽消息命令：读 `references/commands.md`

## 默认工作流

1. 确定日期、工作区、总工时、输出位置；如果用户没指定日期，直接使用当前本地日期。
2. 如果是重新生成日报，先删除目标日期已有日报文件；如果是用户明确要求调整现有日报，才读取旧文件。
3. 用 `state_5.sqlite` 查这一天、这个工作区下的线程，作为候选入口。
4. 再直接扫描 `~/.codex/sessions/{year}/{month}/{day}/` 里目标日期前后至少各 1 天的 `jsonl`，按每条消息的 `timestamp` 判断是否落在目标日期；只要目标日期内有消息、且 `cwd` 是目标工作区或其子目录，就纳入候选。
5. 取出每个候选线程对应的 `rollout_path`。跨天线程要读取目标日期内的消息，同时向前回看同一线程的必要上下文，确认业务目标、分支、文件和最终结论；不要只看目标日期当天截断片段。
6. 先看每个线程目标日期内的首尾时间、真实用户消息数和有效助手消息数，用会话跨度判断投入权重。这里必须使用去重、去上下文噪音后的消息，不要把 `AGENTS.md instructions`、`environment_context`、developer message、系统摘要或重复的 `response_item` 算进去。
7. 对候选工作线程读取完整关键内容，不要只抽用户消息关键词；尤其要看实际修改、验证、提交、推送、合并、发版和最终结论。
8. 运行 token 统计脚本或命令，提取目标日期的 `total_tokens`，按“今日 AI 协作量”格式写在 `# 工作内容` 标题下面；如果找不到或执行失败，写“未统计”。
9. 扫描目标工作区下的 Git 仓库，核对当天提交、未提交改动、当天新增/修改的需求文档、截图、测试记录和代码文件。Git 不是唯一依据，但必须用于校正“今天实际新增/继续推进了什么”。
10. 判断哪些线程属于真实工作，哪些应该排除；如果会话和 Git 互相印证，优先作为主线；如果只有 Git 改动但无会话，要判断它是否是今天工作产物，不能直接忽略。
11. 先列出当天所有明确编号、项目名或工作主线，再归并日报。当天存在多个 Issue/Bug/需求编号时，优先按编号拆分；没有编号支撑的内容默认并入相关主线，避免输出泛泛的“其他事项”。
12. 如果用户环境能查询 Issue/Bug tracker，只读校正类型和进度；否则根据会话、Git、提交、验证和用户确认来判断进度。
13. 写成自然、简短、可读的中文日报。
14. 如果用户同时要求生成周报，或者目标日期是周五且用户没有明确说“不生成周报”，在日报落盘并校正后，可以按“周报生成工作流”汇总本周日报生成一份本地周报。

## 漏项或进度修正工作流

当用户指出“今天不止这些”“某个需求已经上线”“你漏了某条会话/编号”时，不能只增量处理用户点名的那一条。必须重新扫描目标日期内当前工作区所有已更新线程，尤其要重新读取 `state_5.sqlite` 的 `title/updated_at/rollout_path` 和 JSONL 当天新增消息，再按完整候选集重算日报和周报。

修正时必须特别检查：

- 目标日期内 `state_5.sqlite` 标题命中 `编号 + 日期` 的线程，例如 `PROJ-123 2026-06-18`。
- 上次生成日报后仍在继续更新的跨天会话，不能因为前一轮已经扫描过就跳过。
- 同一编号后续又出现测试环境发布、重新合入、重新提审、线上回归或用户明确说“今天已上线”的收口信号。
- 修正文档时要同步重排进度、排序和工时，不能只在原文末尾追加一句。

## 周报生成工作流

当用户说“写周报 / 生成周报”时，默认按下面流程处理：

1. 确认目标周范围；用户未指定时，按当前日期所在自然周处理，周一到周日。
2. 先定位目标工作区里已有日报目录，优先读取该周每天的日报 Markdown 文件，而不是重新从零扫描所有会话。
3. 如果某天日报缺失，再按“默认工作流”补查这一天的会话、Git 和文件变动，生成可用于周报的日级摘要。
4. 合并一份本地周报 Markdown 文件。默认输出到日报目录同级的 `weekly/` 目录；如果项目已有周报目录约定，沿用原约定。
5. 周报按编号或工作主线归并，不要把每天的日报机械拼接；同一主线跨多天推进时合并成一条，并写清楚本周进展和当前结果。
6. 周报默认不写 AI 协作量、本周思考、下周计划、需协调与帮助；只有用户明确要求这些章节时才补。

周五自动周报规则：

- 当目标日期是周五，且本次任务是“生成日报 / 生成今天日报 / 生成日报并周报”，默认在日报完成后同步生成本周周报。
- 周报只基于本周已经落盘的日报文件做汇总；除非某天日报缺失或用户指出漏项/进度错误，否则不要重新扫描所有会话。
- 自动周报只生成本地 Markdown 文件。
- 周报文件默认写到日报目录同级 `weekly/{周一日期}_{周五日期}.md`；如果项目已有其他周报目录或命名约定，沿用已有约定。
- 周报默认只输出 `# 工作内容`；不得额外输出 AI 协作量、思考复盘、下周计划或协调事项，除非用户明确要求。

周报推荐结构：

```md
# 工作内容

**工作项｜1、PROJ-123 报表筛选体验优化｜进度 100%**
汇总本周围绕该工作项完成的关键动作和结果，写清楚当前结果。

**工作项｜2、BUG-456 导出文件名异常修复｜进度 90%**
汇总本周围绕该问题完成的排查、修复、验证和待收口内容。
```

## Token 消耗统计

生成日报时，默认把目标日期的 AI 协作量放在 `# 工作内容` 标题下面。

优先查找或使用下面几类命令：

- 用户明确提供的 token 统计命令
- 当前工作区下的 `script/codex_token_report/codex_token_report.py`
- 当前工作区下的 `scripts/codex_token_report.py`
- 当前工作区文档子目录中的 token 统计脚本

从输出中至少提取这一类信息：

```text
total_tokens: 593,319,807
```

日报里写成：

```md
# 工作内容

今日 AI 协作量：
总消耗：5.93 亿 token
折算工作量：约 5.8 人日
人日拆分：调研排查 1.6、方案文档 1.0、代码实现 1.2、测试回归 1.4、发布协作 0.6
```

如果脚本执行失败，不要阻塞日报生成；在 `# 工作内容` 下面写 `今日 AI 协作量：未统计`，并在最终汇报里说明失败原因。不要把 `input_tokens`、`cached_input_tokens`、`output_tokens` 等明细写进日报正文，除非用户明确要求。

AI 协作量折算规则：

- 展示总消耗时，把 `total_tokens` 四舍五入到“亿 token”两位小数，例如 `792,419,642` 写成 `7.92 亿 token`。
- 不展示缓存 token、输入 token、输出 token 明细。
- 折算工作量采用“动作成本法 + token 强度校正”，不是简单用 token 除以固定数。
- 先按当天日报条目和证据把工作拆到五类：`调研排查`、`方案文档`、`代码实现`、`测试回归`、`发布协作`。
- 每类人日按当天实际动作估算，参考会话跨度、真实消息密度、代码/文档产出、测试/上线证据；每类保留 1 位小数，合计等于“折算工作量”。
- 同一天重复生成时，AI 人日拆分必须稳定，不能因措辞改写随机波动；如果工作证据没有变化，应保持同一类目权重。
- 如果当日大量工作是需求梳理、问题归档、线上数据核查、复现和根因判断，优先加到 `调研排查`。
- 如果当日有方案说明、测试案例、沟通材料或文档沉淀，优先加到 `方案文档`。
- 如果当日有前后端代码改动、脚本修复、提交或评审，优先加到 `代码实现`。
- 如果当日有测试环境复现、真机复测、回归截图、上线后核验，优先加到 `测试回归`。
- 如果当日有合入、构建、发布回复、评审/发布协同，优先加到 `发布协作`。
- AI 折算人日是协作工作量，不等于当天考勤工时；日报条目里的小时数仍按总工时守恒。

## 跨天会话和 Git 校正规则

生成日报时不能只按“线程创建日期”或 `state_5.sqlite.updated_at` 取线程，因为真实工作经常发生在旧会话里继续开发、测试和改文档。

必须同时覆盖三类来源：

- **当天新会话**：目标日期新建或更新的目标工作区会话。
- **跨天续做会话**：前一天或更早创建、但目标日期内仍有用户/助手消息、工具调用、提交、测试或发版动作的会话。
- **Git/文件变动**：目标日期内的提交、合并、推送、未提交代码、需求文档、截图和测试记录。

跨天会话处理方式：

- 以消息 `timestamp` 是否落在目标日期为准，不以 `jsonl` 文件所在日期或线程标题为准。
- 对落在目标日期的片段，要回看同一线程前文，理解“今天继续做的是哪个业务目标”，但日报只计算目标日期内实际推进的部分。
- 如果一个旧线程今天只做很小的收尾，就并入对应主线；如果今天在旧线程里继续写代码、测试、发版，则要按实际投入写进日报。

Git 校正方式：

- 优先扫描目标工作区下一级和二级常见仓库的 `.git`，包括代码仓库和文档仓库。
- 对每个相关仓库看 `git log --all --since/--until`、`git status --short`，必要时看 `git diff --stat` 和当天新增/修改文件。
- 对当天作者是用户本人、或明显来自当前会话操作的提交，要逐条反查是否已经被日报某条主线覆盖；如果没有覆盖，要么并入对应主线一句话，要么说明为什么排除。
- 同一工作项跨多个仓库时，不要只写最显眼的一端；兼容修复、测试环境发布、文档截图更新如果是同一天同一主线的一部分，也要压缩进同一条业务事项。
- Git 只用于校正主线和权重，不要把无关格式化、临时脚本、个人生活文档写进工作日报。
- 如果用户已经明确给了今天主要工作，Git/会话扫描用于补漏和校正表述，不要推翻用户给出的主线。

## 上线或收口日志识别方式

扫描线程标题、真实用户消息、助手结论、工具输出和 Issue/Bug 评论时，如果出现“编号 + 目标日期”的组合，例如目标日期是 `2026-06-18`，会话标题里出现 `PROJ-123 2026-06-18`，就把它识别为该编号在目标日期上线或收口的候选信号。

规则：

- 如果线程标题本身就是或明显以 `编号 YYYY-MM-DD` / `编号 YYYY-M-D` 形式命名，这个日期可视为上线或收口日期。
- 只有日期等于本次日报目标日期时，才写成“当日上线”或“今日已收口”；如果日期不是目标日期，只能作为背景信息。
- 如果 Issue/Bug 评论里有“已上线 / 已修复 / 已关闭 / 已按方案发布 / 可以让用户再看”等表达，该编号进度可写 `100%`。
- 如果只有会话标题强命中但没有外部评论，正文要写“会话标题标记今日收口”或“今日按会话收口”，不要冒写成“平台已回复上线”。
- 同一个编号既有开发/测试动作，又有目标日期上线日志或外部评论时，要把上线结果并入该编号条目，不要拆成单独一条“上线日志”。
- 该规则用于补充结果判断，不替代会话和 Git 对工作内容的分析；如果只出现上线日期标题但没有当天投入，也可以写成该编号的上线结果，但工时要按实际投入给低权重，不要夸大为大量开发工作。

## JSONL 消息提取规则

本地 JSONL 里同一条对话经常会同时出现在 `event_msg` 和 `response_item`，而且 `response_item` 的第一条 user message 往往包含 `AGENTS.md instructions` 和 `<environment_context>`。这些不是当天工作内容，不能直接作为日报素材或消息数。

提取会话上下文时按下面规则执行：

- 优先使用 `event_msg.payload.type in ("user_message", "agent_message")` 作为真实对话流。
- 如果某个旧格式线程没有 `event_msg` 对话流，再回退到 `response_item` 的 user/assistant message。
- 回退读取 `response_item` 时，必须过滤 `# AGENTS.md instructions`、`<environment_context>`、developer/system 指令、工具 schema、模型摘要、`Another language model...` 这类上下文内容。
- 同一时间、同一文本如果同时出现在 `event_msg` 和 `response_item`，只算一次。
- 子 Agent 或 forked context 的 JSONL 可能会在会话开始同一秒重放父会话历史；如果开头 1-2 秒内出现多条连续 user message，要识别为继承上下文，只用于理解背景，不计入当天新工作消息数和权重。
- 当前“生成日报”的线程、子 Agent 生成日报线程、修改日报 skill 的线程，都只能作为本轮工具执行记录，不得写进目标日期工作日报。
- 会话权重只参考清洗后的真实工作消息、工具调用、文件/Git 产出，不参考被污染的原始行数。
- 同一业务目标下启动的多个子 Agent / smoke test / 导出任务，要归并回父级业务主线；不要因为开了多个子线程就拆成多条日报事项或重复计时。

## 线程筛选原则

默认排除：

- 当前这条“让我写日报”的线程
- 任何标题或首条真实用户消息明显是“生成日报 / codex-daily-report / 修改日报 skill / 子 Agent 生成日报”的线程
- 只有测试语句、空消息、随手试命令的线程
- 纯闲聊、纯账号切换、纯工具自检且和当天交付无关的线程
- 用户明确说“不要写进去”的内容

默认保留：

- 需求沟通、方案梳理、页面或流程模拟、业务理解
- Bug 排查、修复、联调、验证、上线准备
- 和实际交付有关的环境搭建、配置接入、测试环境排障
- 用户后续补充说明“这个也算今天工作”的线程
- 子 Agent 或 smoke test 产出的真实交付物，但必须并入对应父级业务主线

## 主线判断规则

写日报前必须先做一次“主线归因”，不要把工具操作或支撑性排查误写成主要工作。

优先级从高到低：

- **真实交付产出**：当天有代码提交、分支推送、合入、测试环境验证、文档沉淀或截图交付的内容，通常是主线。
- **反复迭代验证**：会话里反复出现“改样式、截图、测试、继续、再看、提交、推送、发版”等动作，说明这是实际投入，不要被其中某个排障细节带偏。
- **业务目标表达**：标题和事项要面向团队说明“推进了什么业务功能”，不要用过细的工具名或故障点当标题。
- **联调主线**：如果用户反复围绕某条链路验证，写成“联调与稳定性验证”，不要拆成配置查找、日志排查、环境问题等碎片。
- **支撑性排查**：本地启动参数、构建流程、后端临时报错、账号权限、截图权限等，只有在它们本身是当天交付目标时才单独成项；否则并入对应主线的一句话里。

如果会话内容和 Git 产出不一致：

- 会话说明“反复测试/改样式/联调”，Git 也有相关提交时，以该功能主线为准。
- Git 只有辅助改动但会话主要在业务验证，日报写业务验证，不写辅助改动。
- 只读排查没有形成交付，不要压过当天有提交、有验证的功能主线。

## 归并规则

优先按“真实工作主线”归并，不要按“开了几个会话”归并。

如果当天存在多个需求编号、Issue 编号或 Bug 编号，优先按编号归并，不要为了控制条数把不同编号合并。

优先使用这些粒度：

- 一个需求编号或 Bug 编号
- 一个有稳定内部编号的需求理解或方案输出
- 一个有明确编号的独立问题排查或修复
- 一条完整的联调、验证或接入工作
- 一个明确成体系的业务调研或文档梳理
- 一条从“反复调整、提交、合并、发布、验证”串起来的功能收口工作

不要这样写：

- 按上午、下午拆
- 按线程数逐条罗列
- 把一句话能说清的同类小动作硬拆成很多点
- 把多个编号合成“集中排查”“相关问题处理”
- 把“调内存、查配置、找日志、截图权限”这类支撑动作提升成主线，除非用户当天的真实目标就是它
- 输出 `其他｜`、`杂项｜`、`工作项｜若干问题` 这类兜底标题；如果确实没有编号，也要写清楚具体主线

## 写作要求

- 面向非技术读者，优先写人话；站在团队和管理者视角，写“业务进展”“为什么做”和“交付结果”
- 每条事项尽量控制在一句话，最长不要超过两句；让人一眼看到在做什么、为什么做、工作量大概多少
- 默认每条事项控制在 70 到 90 个中文字符左右；如果超过这个范围，优先删除技术过程和工具细节
- 一条事项里必须能看出背后的目的，例如“沉淀沟通资料”要写明是为了提高多人评审和业务沟通效率
- 不要堆 SQL、接口名、类名、错误码、日志系统名、构建命令、账号名、配置项，除非不写就解释不清
- 技术细节要翻译成业务语言，例如“鉴权失败”可以写成“服务配置问题”，“构建验证”可以写成“多端验证”
- 少写“同步、跟进、处理相关问题”这种空话，尽量写具体动作
- 如果当天有多个 bug，要按不同问题拆开，不要混成一句
- 编号要保留；编号后面的标题必须写成人能看懂的具体业务问题，不要只写编号或系统名
- 标题要高一层，不要用过细的工具名或故障点当标题
- 细节只服务主线：能证明工作质量即可，不要把技术流水账写满
- 如果一条里出现多个专有名词导致非技术人员看不懂，继续向上抽象，保留业务目标和结果即可

## 推荐句式

优先使用下面这种短句式。为了复制到文档后格式稳定，默认不要用 Markdown 列表语法 `-`、`*`、`1.`；工作项统一用中文编号，标题单独一行并整体加粗，正文放在标题下一行。

排序和编号规则：

- 先写需求/功能类，再写 Bug/问题类；两组中间留一个空行。
- 每组各自从 `1` 开始编号。
- 每组内部按进度从高到低排列；进度相同按实际工作量或业务重要性排序。
- 标题格式建议为 `**需求｜1、PROJ-123 标题｜进度 100%｜0.8h**` 或 `**Bug｜1、BUG-456 标题｜进度 90%｜1.2h**`。
- 如果团队不区分需求和 Bug，也可以统一写 `**工作项｜1、PROJ-123 标题｜进度 90%｜1.2h**`。
- 标题里必须包含类型、组内序号、编号和标题、进度、工时；正文末尾不要再重复工时。
- 为兼容 GitHub Markdown 预览，每个条目的加粗标题、正文和下一条标题之间都要用空行隔开；否则会被渲染成同一个段落。
- 不同分组之间同样保留空行，让需求组和 Bug 组在预览里清晰分开。
- AI 协作量块建议用列表展示，并和第一条日报之间留一个空行。

```md
**需求｜1、PROJ-123 报表筛选体验优化｜进度 100%｜2.0h**

复核筛选条件、交互边界和测试口径，完成前端调整、测试环境验证和资料沉淀，当前已进入发布收口。

**需求｜2、PROJ-124 多端提示文案统一｜进度 85%｜1.5h**

梳理多端文案差异，完成主要页面调整和联调验证，剩余边界场景待继续确认。

**Bug｜1、BUG-456 导出文件名异常修复｜进度 90%｜1.2h**

定位特殊字符导致导出失败的问题，完成文件名清洗逻辑和主要回归验证，待最终合入后收口。
```

## 进度判断规则

如果用户没有提供自己的进度口径，可以按下面规则粗略判断：

- 外部平台已关闭、已上线、已修复、已明确无需处理：`100%`
- 目标日期线程标题强命中“编号 + 目标日期”，且上下文指向上线/收口：`100%`
- 已完成代码修复、测试环境验证、文档沉淀，且没有明显待确认事项，但缺少正式上线或关闭证据：`90%`
- 已推送/合入/构建，仍在测试环境回归或上线前收口：`85%`
- 已定位根因并提交修复或形成收窄风险方案，但缺少完整回归、上线或关闭证据：`80%`
- 线上数据已处理但还在等业务/客户确认复现场景：`70%`
- 只完成需求确认、调研归档或方案设计，尚未实现：`50%`
- 如果外部系统有明确进度字段，可作为参考，但要用会话、Git 和评论校正；不要盲目照抄。

## 成稿自检

写完日报后必须自检一遍，再落盘：

- **AI 协作量位置是否正确**：必须先写 `# 工作内容`，下一行写 `今日 AI 协作量：`，再写总消耗、折算工作量和人日拆分；不要展示不必要的 token 明细。
- **是否按主线拆开**：当天出现的每个需求编号 / Bug 编号 / 明确主线是否都有独立事项；如果出现“等问题”“集中排查”“相关问题处理”，要回到会话和 Git 重新拆分。
- **类型是否可靠**：有外部编号的条目是否按真实类型写需求/Bug；不能只按标题猜。
- **进度是否有证据**：`100%` 是否来自上线评论、目标日期线程标题强命中收口日期、关闭/作废/未复现结论；方案项是否保持 `50%` 左右。
- **排序是否正确**：是否先功能类再问题类，中间空一行，各组从 `1` 编号，并按进度从高到低排序。
- **标题格式是否正确**：每条是否是加粗标题单独一行，正文另起一行，标题内是否包含类型、序号、编号标题、进度、工时。
- **是否误用兜底类别**：如果出现“其他/杂项/若干问题”，必须重查编号或具体主线，改为明确事项。
- **空行是否正确**：AI 协作量块后留一个空行；每个条目的标题、正文和下一条标题之间都要留空行，避免 GitHub Markdown 预览挤成一段。
- **是否具体清楚**：每条是否写清楚具体问题、当天动作和当前结果；如果只写“推进、处理、收口”，要补清楚是什么问题。
- **是否剔除非主线支撑项**：资料同步、图床处理、脚本优化、工具能力补齐等不能单独成条，要并入对应主线或删除。
- **业务目的是否清楚**：每条不只写“做了什么”，还要能看出为什么做，例如提升沟通效率、降低发版风险、推动需求确认、收口线上问题。
- **非技术人员是否能看懂**：如果正文里出现多个日志名、错误码、接口名、分支名、构建命令或工具名，要继续压缩成业务表达。
- **工作量是否一眼可见**：保留小时数，且条目数量控制在当天主线能承载的范围；不要为了展示辛苦而堆细节。
- **是否过长**：单条事项如果超过两句，优先删技术过程，保留业务目标、关键动作和结果。
- **是否重复计时**：同一业务目标下的多个子 Agent、smoke test、聊天导出任务，只能归并到一条主线里。
- **是否目的优先**：读者只看标题和第一句话，应能理解这项工作对业务、协作或上线风险的价值。

## 工时拆分

- 总工时默认 `8h`；如果用户明确指定了其他总工时，则以用户指定为准
- 用户在重生成时指定新的总工时，例如“加了很多事，都改为 10h”，必须重新分配所有条目工时，合计精确等于 `10h`，不要只在原 `8h` 版本上局部加时
- 没有多个编号时，一般拆成 3 到 5 条事项；如果当天有多个编号，以“每个编号至少一条”为准，允许超过 5 条
- 各条工时相加必须精确等于总工时；如果删除非主线支撑项，要把它的时间合理分配回实际服务的主线条目
- 工时分配要和事项重量匹配，不要平均分
- 工时权重参考：会话首尾跨度、消息密度、实际代码/文档产出、提交/推送/发布/验证证据
- 如果用户只要内容、不强调工时，可以先按事项写清楚，再决定是否补小时数
- 工时写在标题内，例如 `｜0.8h**`，不要写在正文末尾
- 同一日期重复生成时，工时分配应稳定；如果证据没有明显变化，不要因为标题改写就大幅移动工时
- 大需求持续收口且仍在回归时，不要因为涉及范围大就无限加时；按当天实际投入估算
- 纯方案项通常 `0.8h` 到 `0.9h`；不要因为描述更简短就压到 `0.5h`，除非当天只是很小的补充
- 已上线或已关闭的小项可低于 `1h`；测试环境修复、归档和验证完整的 Bug 通常 `1.0h` 到 `1.2h`

## 推荐格式

```md
# 工作内容

今日 AI 协作量：

- 总消耗：7.92 亿 token
- 折算工作量：约 7.6 人日
- 人日拆分：调研排查 2.1、方案文档 1.5、代码实现 1.6、测试回归 1.8、发布协作 0.6

**需求｜1、PROJ-123 报表筛选体验优化｜进度 100%｜2.0h**

说明具体问题/目标，完成了哪些动作，并形成什么结果。

**需求｜2、PROJ-124 多端提示文案统一｜进度 85%｜1.5h**

说明具体问题/目标，完成了哪些动作，并形成什么结果。

**Bug｜1、BUG-456 导出文件名异常修复｜进度 90%｜1.2h**

说明具体问题/目标，完成了哪些动作，并形成什么结果。
```

默认只输出 `# 工作内容`，其下第一块写 `今日 AI 协作量`，不要额外加 `# 明日计划` 或无关部分。

## 关键边界

- 这套流程的主依据是“Codex 会话记录 + Git/文件变动交叉验证”，不是单看其中一种。
- Git、文件改动、已有日报文件用于补漏和校正，不要反过来靠 Git 臆测整天的工作；但如果存在跨天会话或会话很多很碎，必须用当天提交/推送/文件变动校正主线权重。
- 如果会话内容和代码状态有出入，优先根据用户当天的真实对话判断，再标记必要的推断。
- 如果当前工作区和输出目录都不明确，先优先使用当前工作区；仍然不清楚时，再让用户补充。
- 公开仓库不应保存公司名、内部系统名、真实需求编号、客户信息、固定个人路径、真实账号或敏感截图。

- 需要查线程、查 `rollout_path`、抽用户消息时，优先复用参考命令，不要临时手写一套
